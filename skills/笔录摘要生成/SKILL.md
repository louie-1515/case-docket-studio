# 笔录摘要生成

为案件中的每份笔录生成结构化AI摘要，聚焦事实内容，写入案件JSON的"笔录摘要"字段。

## 触发关键词

- `"生成笔录摘要"`、`"笔录摘要"`、`"摘要生成"`
- `"AI摘要"`、`"笔录总结"`、`"提炼笔录"`
- `"案件摘要"`、`"笔录要点"`

## 前置条件

- 案件JSON已存在于 `<项目根目录>\project\data\{案件名}.json`
- JSON中每条笔录已有"全文内容"字段（MinerU解析后的markdown文本）
- 建议先完成 `/导入案件笔录检索app` 再执行本skill
- 如已有起诉意见书/起诉书，**必须先读取其内容并确定委托当事人与涉案核心人物**，再开始摘要

## 核心原则

### 先定委托当事人，再做摘要

**这是硬性前置步骤，不可跳过。**

每个子agent在处理自己分配的笔录前，必须先读取以下上下文：
1. 起诉意见书/起诉书原文或简要版
2. 案件中的 `委托当事人`
3. 案件中的 `涉案核心人物`（如有）

如果案件JSON中还没有 `委托当事人`，必须先询问用户确认谁是委托我们进行辩护的人，再写入案件JSON顶层。

`涉案核心人物` 可根据起诉意见书/起诉书中的涉案对象补充：

```json
{
  "委托当事人": ["张三"],
  "涉案核心人物": ["王五", "赵六", "李四", "陶某", "李某"]
}
```

### 摘要聚焦规则

- 一切摘要都必须围绕**与委托当事人有关**的事实展开
- 与委托当事人无关的枝节信息，只在确有证明意义时简要保留
- 如果一份笔录主要谈论非委托当事人、且与委托当事人关联极弱，摘要中必须明确其与委托当事人的关联点，而不是孤立复述
- 任何“关键事实”判断，都要以起诉意见书/起诉书中的案件主线为参照

## 数据格式

摘要存储在案件JSON的顶层字段 `笔录摘要` 中，以笔录ID（字符串）为键：

```json
{
  "案件名称": "...",
  "笔录列表": [{"id": 1, "姓名": "张三", ...}, ...],
  "笔录摘要": {
    "1": {"content": "张三首次讯问中交代...", "generated_at": "2026-05-07T..."},
    "2": {"content": "李四询问笔录中承认...", "generated_at": "2026-05-07T..."}
  }
}
```

**注意：** `笔录摘要` 是顶层字段，不是每条笔录的子字段。前端通过 `data["笔录摘要"][str(record_id)]` 获取。

## 工作流程

### 步骤1：确认案件名

询问用户要处理的案件名称，对应data目录下的JSON文件名（不含 `.json` 后缀）。

### 步骤2：读取案件JSON

读取 `<项目根目录>\project\data\{案件名}.json`，获取 `笔录列表`。

### 步骤2.5：读取起诉意见书并确认委托当事人

读取案件JSON中的 `起诉书` 字段，提取：
- 涉案对象列表
- 案件主线事实
- 指控罪名

如果 `委托当事人` 缺失：
- 必须先询问用户确认委托我们进行辩护的人
- 写回案件JSON顶层 `委托当事人`

如果 `涉案核心人物` 缺失：
- 优先用起诉书中的 `structured.当事人`
- 再剔除 `委托当事人`
- 写回案件JSON顶层 `涉案核心人物`

所有后续子agent都必须收到以下共享上下文：

```text
【案件主线】{起诉意见书案件事实简要版}
【委托当事人】{entrusted_parties}
【涉案核心人物】{related_people}
```

### 步骤3：逐份笔录生成摘要

**重要：必须使用多agent并行处理。** 笔录数量通常超过50份，单线程处理极慢。

**策略：**
- 笔录数量 ≤ 20：逐份生成
- 笔录数量 > 20：使用多个executor agent并行处理，按工作量分割
- **工作量分割原则**：不要只按任务类型分agent，还要按工作量分割。例如155份笔录应拆分为3-4个agent，每个处理40-50份
- 启动agent时使用 `mode: bypassPermissions`
- **Fallback**：如果agent因权限失败，改用Python脚本直接处理——写入临时.py文件然后用 `py` 命令执行。脚本方式更可靠，推荐优先使用

**Prompt模板（每份笔录）：**

```
请为以下刑事笔录生成结构化摘要，控制在200字以内：

【案件主线】
{起诉意见书简要版}

【委托当事人】
{entrusted_parties}

【涉案核心人物】
{related_people}

【笔录信息】
- 被讯问/询问人：{姓名}
- 笔录类型：{笔录类型}
- 日期：{日期}

【笔录全文】
{全文内容}

【摘要要求】
1. 核心事实（2-3句）：该笔录中此人交代的关键事实
2. 时间线：提及的关键时间节点
3. 涉及人员：除被讯问/询问人外，提及的其他人员
4. 涉案金额/数量（如有）
5. 态度/立场：配合、抗拒、翻供、回避等

要求：
- **先判断本份笔录与委托当事人的关联强弱**
- **摘要必须围绕对委托当事人有何证明意义来组织**
- 如果笔录主要涉及非委托当事人，必须交代其与委托当事人、哪一事实链条相关
- 必须优先回答：这份笔录对委托当事人的定罪、量刑、角色认定、主从犯判断、金额认定是否有帮助或不利
- **必须跳过个人信息段**（姓名、年龄、身份证、住址、家庭情况、简历、前科等）
- **必须跳过程序性内容**（权利义务告知、申请回避、聘请律师、签字确认等）
- **必须跳过结尾确认段**（"属实"、签名、核对意见等）
- **只聚焦案件事实部分**：关于案情的问答内容
- 聚焦案件事实，不包含任何程序性内容
- 使用第三人称客观叙述
- 控制在200字以内
- 输出格式为纯文本段落，不要分点编号
```

### 步骤4：写入JSON

将生成的摘要写入案件JSON的顶层 `笔录摘要` 字段，保存回文件。

```python
def save_summaries(case_data, summaries, json_path):
    """将摘要写入案件JSON并保存。

    Args:
        case_data: 案件JSON数据
        summaries: {str(笔录id): {"content": 摘要, "generated_at": 时间}} 的字典
        json_path: JSON文件路径
    """
    case_data['笔录摘要'] = summaries

    with open(json_path, 'w', encoding='utf-8') as f:
        json.dump(case_data, f, ensure_ascii=False, indent=2)

    return json_path
```

### 步骤5：汇报结果

向用户汇报：
- 处理笔录总数
- 成功生成摘要数
- 失败/跳过数（如有）
- 示例摘要1-2条

## 摘要质量标准

| 维度 | 要求 |
|------|------|
| 字数 | ≤ 200字 |
| 人称 | 第三人称客观叙述 |
| 内容 | 聚焦事实，排除程序性内容 |
| 完整性 | 覆盖核心事实、时间、人员、金额、态度 |
| 准确性 | 不添加笔录中未提及的信息 |
