# 案情图谱生成

基于案件JSON中的笔录摘要、人物摘要和起诉书内容，提取实体（人物、组织、账户、事件）和关系，生成结构化案情图谱数据。

本 skill 需要同时服务两种图示：
- app 内的关系网络图：适合快速查看节点连接和筛选人物。
- drawio 风格办案导图：适合律师阅卷、汇报和手工精修，形式接近方框主体关系图。

## 触发关键词

- `"生成案情图谱"`、`"案情图谱"`、`"案件图谱"`
- `"实体关系"`、`"关系网络"`、`"案情可视化"`
- `"图谱数据"`、`"案件网络"`、`"人物关系图"`

## 前置条件

- 案件JSON已存在于 `<项目根目录>\project\data\{案件名}.json`
- 建议先完成 `/笔录摘要生成` 和 `/人物摘要生成`
- 如有起诉书内容（由 `/刑事笔录分卷` 阶段4生成），优先使用
- **必须先确认委托当事人与涉案核心人物，再生成图谱**

## 核心原则

### 一切图谱先围绕委托当事人展开

- 图谱不是“把所有名字都画上去”
- 先确定委托当事人，再决定哪些人物、组织、账户与委托当事人存在直接或重要关联
- 非委托当事人节点只有在能够解释委托当事人的资金、指挥、交易、转运、证明关系时才进入主图
- 图谱不得直接写“主犯”“从犯”“首要分子”等法律评价结论，只描述事实作用，例如“组织作用”“联络作用”“转运作用”“收款作用”“辅助搬运”“证言证明”
- 主线必须是人物和主体，事件只作为关系线上的标签、证据或说明，不单独做成方块节点

### 优先产出办案导图需要的数据

生成节点和关系时，要考虑 drawio 导图的可读性：

- 节点名称要短，适合作为方框标题
- `subtype` 要能作为方框副标题，例如“上家”“转运人员”“收款账户”“联络人员”
- 关系标签要短，适合写在箭头上，例如“指派”“采购”“转运”“收款”“证明”
- 不同流程必须通过关系线区分，而不是混在同一种线里。优先使用 `flow` 字段：`货物流`、`资金流`、`指挥联络`、`证据关系`、`事实关系`
- `style="solid"` 表示明确证据支持的直接关系
- `style="dashed"` 表示间接、推断、指示、介绍等关系
- 线条必须适合 drawio 折线图展示，关系方向要清楚，不生成依赖曲线才能理解的复杂布局
- 每条关系仍必须保留 `evidence`，供后续复核和导图精修

## 工作流程

### 步骤1：确认案件名

询问用户要处理的案件名称。

### 步骤2：读取案件数据

读取案件JSON，提取：
- 笔录列表（所有人物和笔录摘要）
- 人物摘要（如已生成）
- 起诉书内容（如有）
- 委托当事人 / 涉案核心人物（如有）

同时读取起诉意见书全文（如MinerU解析结果存在）：
`E:\Mineru\解析结果\起诉意见书.pdf-{uuid}\full.md`

### 步骤3：先确认委托当事人，再选择重要角色

先读取起诉意见书中的涉案对象，并与案件JSON中的 `委托当事人`、`涉案核心人物` 对齐。

然后向用户展示：
- 委托当事人
- 涉案核心人物（默认推荐）
- 其他高关联人物

让用户确认"重要角色"（将作为独立节点）。

### 步骤4：实体和关系提取

**重要：必须使用多agent并行处理。** 图谱生成需要分析大量数据，单线程处理极慢。

**策略：**
- 使用 executor agent 并行处理
- 启动agent时使用 `mode: bypassPermissions` 以避免文件读写权限被拒

**Prompt模板：**

```
请基于以下案件数据，提取案情图谱的实体（nodes）和关系（edges）。

【案件名称】{案件名称}
【案件主线】{起诉意见书简要版}
【委托当事人】{entrusted_parties}
【涉案核心人物】{related_people}
【重要角色（用户选择）】{selected_parties}
【人物摘要】{person_summaries}
【笔录摘要】{record_summaries}
【起诉书内容】{indictment}

【实体提取要求】
提取以下类型节点，总数量控制在20个以内。节点以人物/主体为主，不把事件单独做成节点：
1. 人物节点（type="person"）：
   - 重要角色：importance="primary"，独立成节点
   - 次要角色：可考虑聚合到组织节点或作为importance="secondary"
   - subtype：只能写事实作用，例如“委托人”“上家”“下家”“转运人员”“联络人员”“证人”“收款人员”
   - 禁止在 label、subtype、description 中直接写“主犯”“从犯”“首要分子”等结论性法律评价
   - description 应包含：身份背景、事实作用、关键行为、证据来源提示（如有）

2. 组织/公司节点（type="organization"）：
   - 如XX公司、犯罪团伙等
   - subtype：公司、团伙、机构等

3. 账户节点（type="account"）：
   - 如个人账户、仓库、车辆等
   - subtype：个人账户、仓库、车辆等

4. 事件处理规则：
   - 原则上不生成 `type="event"` 节点
   - 关键交易、会面、抓捕、接驳、转运等事件，应写入关系的 `label`、`flow`、`evidence` 或节点 `description`
   - 只有当某一事件本身是阅卷必须独立讨论的争议焦点时，才可例外生成事件节点

【关系提取要求】
1. 直接关系（type="direct", style="solid"）：有明确笔录或起诉书记载的
2. 间接关系（type="indirect", style="dashed"）：推断或间接证据支持的
3. 关系标签要简洁（如"伙同作案"、"指派"、"销售"、"采购"、"运输"）
4. 每条关系必须有 `flow` 字段：
   - `货物流`：采购、销售、运输、转运、接驳、仓储、交付
   - `资金流`：付款、收款、转账、定金、账户、结算
   - `指挥联络`：指派、安排、介绍、联系、通知、沟通
   - `证据关系`：供述、辨认、证实、印证、证明
   - `事实关系`：其他无法归入上述流程的事实联系
5. 每条关系必须有证据来源（引用具体笔录或起诉书段落）

【输出格式】
输出严格的JSON格式：
{
  "version": "1.0",
  "generated_at": "ISO时间",
  "selected_parties": ["用户选择的重要角色"],
  "nodes": [
    {
      "id": "node1",
      "label": "显示名称",
      "type": "person|organization|account|event",
      "subtype": "委托人|上家|下家|转运人员|联络人员|证人|公司|仓库|账户",
      "description": "详细描述（含身份背景、事实作用、关键行为、证据来源提示）",
      "importance": "primary|secondary",
      "members": [],
      "records": []
    }
  ],
  "edges": [
    {
      "id": "edge1",
      "source": "node1",
      "target": "node2",
      "label": "关系描述",
      "type": "direct|indirect",
      "style": "solid|dashed",
      "flow": "货物流|资金流|指挥联络|证据关系|事实关系",
      "evidence": "证据来源"
    }
  ]
}

要求：
- 先围绕委托当事人构建主干关系，再补充高关联节点
- 图谱应优先服务于辩护分析，而不是平均呈现全案
- 节点总数 ≤ 20
- 关系必须有证据来源
- 关系必须有 flow，用不同流程线展示货物流、资金流、指挥联络、证据关系
- id使用node1, node2...和edge1, edge2...格式
- members字段用于组织节点包含的人物列表
- records字段用于节点相关的笔录id列表（如有）
- description要丰富：人物节点应包含身份背景、事实作用、关键行为、证据来源提示
- 不得使用“主犯”“从犯”等结论性法律评价，不得把事件作为主线方块
```

### 步骤5：写入JSON

将生成的图谱数据写入案件JSON的顶层 `案情图谱` 字段，保存回文件。

### 步骤6：汇报结果

向用户汇报：
- 节点总数及各类节点数量
- 关系总数及直接关系/间接关系数量
- 重要角色节点列表
- 提示用户可重新选择重要角色生成不同侧重点的图谱

## drawio 导出适配

app 会根据 `案情图谱.nodes` 和 `案情图谱.edges` 自动导出 `.drawio` 文件。导出时遵循以下视觉约定：

- `importance="primary"` 的节点作为导图主干节点，优先放在中轴
- 委托当事人必须是主干节点
- 人物节点使用方框，组织、账户用不同底色区分，事件原则上不单独成框
- `style="solid"` 导出为粗实线箭头
- `style="dashed"` 导出为虚线箭头
- `flow` 导出为不同颜色的关系线，用于区分货物流、资金流、指挥联络、证据关系
- 关系线必须使用直线或折线，不使用曲线
- 边的 `label` 会显示在箭头附近

因此生成图谱数据时，不要把长段事实塞进 `label`，长事实放入 `description` 或 `evidence`。

## 节点类型说明

| type | subtype示例 | 说明 |
|------|------------|------|
| person | 委托人、上家、下家、转运人员、联络人员、证人 | 案件中的人物及其事实作用 |
| organization | 公司、团伙、机构 | 组织实体 |
| account | 个人账户、仓库、车辆 | 资金账户/资产 |
| event | 原则上不使用 | 事件应作为关系线和证据说明，不作为主线节点 |

## 重新生成支持

用户可选择不同的人物组合重新生成图谱。
